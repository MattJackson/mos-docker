# Incident: 256 GB macOS install destroyed by auto-install-mode (2026-05-06)

## Summary

A 256 GB sparse-allocated macOS install (`/data/macos/mac_hdd_ng.img`,
weeks of state) was overwritten with a fresh 1 MiB blank file at
`2026-05-06 00:32:54`, during a session refactoring the docker-compose
phase chain. Original install was unrecoverable — no backup existed.

## Mechanism

The Phase 1/2/3 launcher scripts each contained an "install mode"
branch:

```bash
CURRENT_SIZE=$(stat -Lc%s "${IMAGE_PATH}" 2>/dev/null || echo 0)
if [ "${CURRENT_SIZE}" -lt 1048576 ]; then
    qemu-img create -f raw "${IMAGE_PATH}" "${DISK_SIZE:-256G}"
fi
```

Three failure modes:

1. **`stat` failure swallowed by `|| echo 0`.** Any transient failure
   (filesystem hiccup, bind-mount race, sparse-file metadata edge case)
   silently turned size into 0, taking the install path. `qemu-img
   create -f raw` does not refuse to overwrite an existing file.

2. **Symlink redirection.** `volumes/disk.img` was a symlink to
   `/data/macos/mac_hdd_ng.img`. Any host-side process that did
   `: >volumes/disk.img` (truncate via shell redirect) — e.g. the old
   `setup.sh` — followed the symlink and zeroed the real file.

3. **Implicit, undocumented destructive behavior.** The launcher
   decided "install mode" automatically based on a fragile size check.
   No user opt-in. No confirmation. No marker file. The same script
   that boots production decided to wipe the disk under conditions a
   normal operator wouldn't know about.

The actual triggering event is forensically unclear (file owner went
from `root:root` to `matthew:matthew`, timestamps point at 00:32 on
2026-05-06). What's certain is that one of the three modes fired and
the launcher silently called `qemu-img create -f raw $FILE 256G` — or
something else (e.g. `setup.sh`-like logic in agent code) ran
`qemu-img create -f raw $FILE 1M` — either way, `disk.img` was
truncated and the install gone.

## Cost

- 250+ GB sparse macOS install destroyed (estimated ~30 GB actual
  data)
- Hours of iteration to figure out what happened, what could be
  recovered, and how to redesign so this can't happen again
- Forced full reinstall workflow rebuild

## Fix

The fix is multilayered, applied in commit (this push):

1. **Eliminate `|| echo 0`.** `stat` is allowed to fail loud now;
   missing file ≠ "size 0, please overwrite".
2. **Install mode is no longer automatic.** Requires explicit
   `MOS_INSTALL_MODE=1` env var (or running the dedicated `install`
   sub-command which sets it).
3. **`run.sh` (production) NEVER calls `qemu-img create` on the data
   disk under any circumstances.** That code path simply doesn't
   exist anymore. Production only reads.
4. **`install.sh` refuses to overwrite an existing `>1 MiB`
   `disk.img`.** Even when explicitly invoked. To reinstall, the
   operator deletes the file manually first — that's the consent
   gesture.
5. **Single launch script.** Five copies of the bug were patched
   today. The architecture collapses to one launcher
   (`scripts/run.sh`) + a separate install script (`scripts/install.sh`)
   + a separate test runner (`scripts/test.sh`). Bug fixes happen in
   one place.
6. **No more host-side `setup.sh`** that touches `volumes/`. The
   container manages `/data/` directly. No host-side scripts
   redirecting through symlinks into bind mounts.

## Standing rule

Anything in this codebase that would create/wipe a `disk.img`
must:

- Require an explicit env-var or sub-command opt-in (no automatic
  inference from size or presence)
- Refuse to operate if the target already looks like a real install
  (>1 MiB)
- Print a 5–10 second abort window before wiping
- Sit in code that is NEVER reached by the default `docker compose
  up` / `docker run` path

Any future code review that adds an "auto-detect install vs boot
mode" branch should reject the change.

## Lessons applicable beyond this codebase

- **Implicit destructive operations are bombs with random fuses.**
  An operation can be safe under every circumstance the author
  imagined and unsafe under one they didn't. Always require explicit
  consent for destruction.
- **Auto-magic falls back to default values when input is wrong, not
  to "stop and ask".** The pattern `$(cmd 2>/dev/null || echo 0)` is
  almost always wrong if `0` triggers a different behavior than the
  real value would have.
- **Bind-mount + symlink redirection is a footgun.** A host-side
  truncate of `volumes/disk.img` followed the symlink and reached the
  real 256 GB file.
- **One source of truth for any destructive logic.** When the bug
  exists in 5 launcher scripts, you fix it 5 times. Or you fix it
  4 times.
